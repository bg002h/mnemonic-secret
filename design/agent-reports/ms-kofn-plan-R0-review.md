## Critical

### C1 — `combine_shares` returns a garbage `Tag` (random share-set id), not the type tag; the SPEC signature `-> Result<(Tag, Payload)>` is under-defined for shares and the plan routes blindly through `discriminate`.

Task 1.4 Step 3 (plan:124–125) says: "route `secret` through the existing `discriminate(&secret)` → `(Tag, Payload)`". Trace the recovered secret-at-S through `discriminate` (`envelope.rs:95–155`):

- A K-of-N share-set's `id` is a **random 4-char codex32 id** (SPEC §1:16, §2 step 1; plan Task 1.3:109). The recovered secret-at-S therefore has id = that random string (e.g. `q9x8`), NOT `entr`/`mnem`.
- `discriminate` builds the returned Tag via `Tag::try_new(tag_str)` on the id field (`envelope.rs:117–120`). Because the random id is drawn from the codex32 charset, `try_new` **succeeds** (`tag.rs:27–46` only alphabet-checks) and returns `Tag(random_id)`.
- So `discriminate` returns `(Tag=<random id>, payload)`. `combine_shares` would then return that random-id Tag to the caller.

This is wrong. The payload-KIND (entr vs mnem) is recovered from the **prefix byte** (0x00/0x02) of `data()`, which `discriminate`'s prefix-dispatch handles correctly — but the **`Tag`** in v0.2-shares is NOT a type tag (SPEC §1:16 "NOT a type tag in v0.2-shares"); it is the random set id. The canonical single-string `decode` returns `Tag::ENTR`; `combine_shares` of an entr share-set must likewise return `Tag::ENTR` (and a mnem share-set → also `Tag::ENTR`, since mnem rides under the entr tag per `package(Tag::ENTR, Payload::Mnem)` — verified `encode.rs:91–101` always uses `Tag::ENTR`). The recovered random id must be **discarded** (or surfaced only as an informational `id` field on the CLI, never as the returned `Tag`).

Why this matters concretely:
- Returning `Tag(random_id)` means downstream `ms combine`/`ms-shares combine` would carry a meaningless tag; any code that asserts `tag == Tag::ENTR` (e.g. a re-encode of the recovered secret, or the toolkit feeding `combine --to ms1` into `bundle`) gets the wrong tag and would mis-encode or reject.
- Worse: if the plan's "route through `discriminate`" is taken literally AND the threshold gate is bypassed (per Task 1.4's flag) but the rest of `discriminate` runs unchanged, the returned Tag is the random id — silently wrong, no error. This is the kind of garbage-output bug the C1-class review exists to catch.

**Fix (pin in Task 1.4 + reconcile SPEC §2 `combine_shares`):** `combine_shares` MUST return `Tag::ENTR` for the recovered secret (the canonical entr type tag, matching how `encode`/`package` always tag ms1 payloads), NOT the share-set's random id. State this explicitly. The cleanest path (see I1) is for `combine_shares` to call a prefix-only dispatch helper that returns just the `Payload` from `data()`, then pair it with `Tag::ENTR` itself — it must NOT lift the id field as the Tag. Add a round-trip test asserting `combine_shares(split(ENTR, …))?.0 == Tag::ENTR` (the random id is re-rolled per `encode_shares` call, so the returned tag being `entr` regardless of id is the load-bearing assertion). This also retires the ambiguity in SPEC §2:68 ("route through the existing `discriminate()`") — `discriminate` returns the id-as-Tag, which is wrong for shares.

### C2 — the m3 "wildcard swallows the new variants" fold is applied ONLY to the toolkit (Task 3.3); the IDENTICAL wildcard in **ms-cli's** `From<ms_codec::Error>` is unaddressed — and it breaks Task 4.1's own contract-test re-spec.

## Important

### I1 — Task 1.4's `discriminate`/threshold-gate factoring is genuinely hand-wavy ("factor the prefix-dispatch out … or pass a flag"); pin the exact approach.

The plan self-flags this (plan:222) and offers two non-committal options. The reviewer asked for the cleanest factoring; here it is, verified against `envelope.rs:95–155`:

**Pinned approach — extract a prefix-only payload dispatcher; do NOT add a flag, do NOT reuse `discriminate` for shares.**

`discriminate` does two separable jobs: (A) wire-invariant gates on the *string header* (HRP 100–103, threshold 105–109, share-index 110–114) + Tag construction from the id (117–120); (B) payload-kind dispatch on the *recovered bytes* (`c.parts().data()` 131, prefix match 134–152). For `combine_shares` we want ONLY (B), and we want the Tag to be `Tag::ENTR` (C1), not the id.

Factor (B) into:
```rust
// envelope.rs — operates on already-recovered payload bytes, no header gates.
pub(crate) fn dispatch_payload(data: &[u8]) -> Result<Payload> {
    match data.first() {
        Some(&RESERVED_PREFIX) => Ok(Payload::Entr(data[1..].to_vec())),
        Some(&MNEM_PREFIX) => {
            let language = data[1]; let entropy = data[2..].to_vec();
            let p = Payload::Mnem { language, entropy }; p.validate()?; p_ok(p)
        }
        Some(&other) => Err(Error::ReservedPrefixViolation { got: other }),
        None => Err(Error::ReservedPrefixViolation { got: 0 }), // defensive
    }
}
```
Then:
- `discriminate` (the single-string decode path) keeps its header gates (A) — with the threshold gate at 105 RELAXED per Task 1.5/I1 — and its tail becomes `dispatch_payload(&c.parts().data())` returning the payload; it still pairs with the id-Tag (correct for v0.1 single-strings where id IS the type tag).
- `combine_shares` calls `interpolate_at(&parsed, Fe::S)?` → `let bytes = secret.parts().data();` → `let payload = dispatch_payload(&bytes)?;` → returns `(Tag::ENTR, payload)`. It NEVER touches `discriminate`'s header gates (no threshold gate to fire) and NEVER lifts the random id as a Tag (closes C1).

This is cleaner than "pass a flag" (a `skip_threshold_gate: bool` would also have to skip the id-as-Tag lift to fix C1, making the flag carry two unrelated meanings) and cleaner than duplicating the prefix match. The plan must replace plan:124–125's "route through `discriminate`" with this. (Bounds note: `dispatch_payload` indexing `data[1]`/`data[2..]` is safe because the recovered secret is `[prefix]||payload` with payload ≥ 16 B — but add the `data.len()` guard for the mnem branch to avoid a panic on a malformed-length recovery; the secret-at-S length is fixed by construction, so this is belt-and-suspenders.)
