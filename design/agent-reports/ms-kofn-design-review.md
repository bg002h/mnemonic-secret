# K-of-N design review (pre-SPEC) — SOUND-WITH-CHANGES

opus architect, design-direction review BEFORE the SPEC (a formal SPEC R0 follows). Crypto verified directly against codex32 0.1.0 source + runtime spikes (entr+mnem all 5 lengths, k∈2..9, n up to 31, the short-circuit hazard, every mismatch error). entr+mnem sharing genuinely works; threshold-field pivot is right; byte-identity holds. **2 Critical** + 7 Important folds the recon/prior-R0 did NOT surface — the C2/I2/I3 corrections from `ms-v0-2-spec-R0-review.md` are necessary but NOT sufficient.

## Critical (design-breaking — must change before SPEC)

### C1 — `combine_shares` MUST reject any input share at index `s`; the secret-at-S short-circuit silently bypasses ALL validation
`interpolate_at` (`lib.rs:259-263`) returns the first input share whose index == target **directly, before** the Lagrange loop that detects `RepeatedIndex`. `combine_shares` calls `interpolate_at(_, Fe::S)` → target IS `s`. Spike: `interpolate_at([secret_at_S, anything], Fe::S)` → `Ok`, short-circuits to the secret, the other shares never validated; `[secret_at_S, secret_at_S]` → `Ok`, no `RepeatedIndex`. codex32 does NOT reject a threshold∈2..9 string with index `s` (the `parts_inner` guard `lib.rs:202` only fires for threshold==0) — so a secret-at-S is a structurally valid, parseable string a user can paste into `combine`. **Fix:** `combine_shares` pre-check (before `interpolate_at`) MUST reject any input share whose index is `s` (new `Error::SecretShareSuppliedToCombine` or reuse `ShareIndexNotSecret`), IN ADDITION to I3's distinct-index + count≥k. Not identified by the recon.

### C2 — the toolkit must NOT add a bare `mnemonic combine`; it collides with two existing combine verbs
`mnemonic-toolkit/src/main.rs:110-119` already ships `seed-xor` (split-OR-combine, one subcommand) and `slip39` (split-OR-combine K-of-N, one subcommand) — the established **split-or-combine-in-one-subcommand** pattern. A bare `mnemonic combine <share>...` is ambiguous (combine *what* — XOR / SLIP-39 / codex32 shares?) and breaks the pattern. **Fix:** mirror the pattern: either (a) a single disambiguated `mnemonic ms-shares` / `codex32-shares` subcommand with split/combine modes, or (b) combine-only with an unambiguous name (`mnemonic ms-combine`) or fold into `convert --from ms1-shares=…`. Also decide: does the toolkit get split at all (redundant with `ms split` in ms-cli) or combine-only? SPEC must pick a name that doesn't read as a third generic "combine."

## Important (fold into SPEC)

- **I1 — `ms decode` of a lone share must return a distinct, actionable error**, not route into `discriminate()`. A lone distributed share parses cleanly (threshold char, index, `data()[0]` = interpolated garbage e.g. 0xec); if it reaches `discriminate()` it returns a misleading `ReservedPrefixViolation{got:0xec}`. Fix: single-string `decode` dispatches on threshold FIRST: 0 → discriminate path; 2..9 → new `Error::IsShareNotSingleString{threshold,index}` (Display points to `ms combine`). The toolkit's `decode_card` (`cmd/inspect.rs:171`) propagates via `?` → surfaces cleanly. `ms inspect` of a lone share should be FIRST-CLASS (report threshold/id/index structural fields), distinct from decode.
- **I2 — surface `MismatchedHrp` + `MismatchedLength`** (omitted in old §3). All five fire eagerly from `interpolate_at` (spike-confirmed): `ThresholdNotPassed`, `MismatchedThreshold/Id/Length`, `RepeatedIndex`. `combine_shares` returns them as `Error::Codex32(inner)` (`error.rs:11,146-150`); `codex32_friendly.rs:40-48` (ms-cli) already maps them — ensure the toolkit's `friendly.rs` does too (verify or add).
- **I3 — `encode_shares` signature is `(tag, Threshold, n, &Payload)`, NOT `(tag, Threshold, payload_set)`** (the recon/MIGRATION inv-4/old-§5 carry the stale `payload_set` form — incompatible with "derive all N internally"). Caller gives ONE secret + count n; ZERO case is `encode_shares(tag, ZERO, 1, &p)` ≡ `encode(tag,&p)` (spike reproduced `ms10entrsqqqq…`). Internally: secret-at-S via `from_seed(HRP,k,id,Fe::S,[prefix]||payload)`; k−1 random defining shares at distinct non-S indices, CSPRNG payloads same byte-length (use `getrandom` directly — `mk_codec::encode` precedent, no injected-RNG param); `interpolate_at` for the remaining n−(k−1). **Amend MIGRATION.md + §5 inv 4 to the new signature** — the recon's "inv-4 survives unchanged" is WRONG (signature changes; byte-identity claim survives).
- **I4 — `n` bound is exactly `2 ≤ k ≤ n ≤ 31`** (spike: 31 non-S indices `qpzry9x8gf2tvdw03jn54khce6mua7l`; n=32 exhausts) — hard-validate with a clean error, not a codex32 panic. Anti-collision `id` blocklist must STILL include `mnem` (now emit-allowed but a random `id` colliding with entr/seed/xprv/mnem/prvk must re-roll): introduce a SEPARATE `RESERVED_ID_BLOCKLIST = {entr,seed,xprv,mnem,prvk}` distinct from the decoder-reject `RESERVED_NOT_EMITTED_V01 = [seed,xprv,prvk]` (`consts.rs:62`, mnem removed in Cycle 1). Re-grep names at SPEC time.
- **I5 — relaxing the threshold-0 reject breaks an existing CONTRACT test** `crates/ms-cli/tests/decode_rejects_threshold_not_zero.rs` (+ possibly `inspect_non_zero_prefix.rs`). Per the shared-string-change sweep lesson: grep ALL tests + re-spec these (lone share now → `IsShareNotSingleString`, not `ThresholdNotZero`). `envelope.rs:105-108` + SPEC §4 rule 3 reframed reject→route.
- **I6 — §5/MIGRATION amendment is wider than inv 1+2:** also the §5 registry table (`0x01=v0.2-entr-share` `SPEC:224` → `0x00=entr,0x01=unallocated,0x02=mnem,0x03+=unallocated`; make explicit that prefix-byte axis ⊥ threshold axis), §4 rules 3/4 (reject→route), §8 deferred-table row ("K-of-N for entr" → shipped, broaden to entr+mnem).
- **I7 — `--json` wire-shape additions (inspect share fields; combine shape) are NOT `schema_mirror`-gated** (flag-NAME only). `ms split`/`ms combine` subcommand NAMES ARE caught (reflective gui-schema). The `--json` fields must self-update `mnemonic-gui/src/schema/ms.rs` + manual `43-ms.md` via the paired-PR leading discipline.
