# Phase 1 R0 review — ms mnem — round 0

**Verdict:** GREEN (0C / 0I)

The codec layer is implemented correctly and matches the plan's Phase-1 contract (Tasks 1.1–1.4). The two hard gates — entr byte-identity and a hand-computed wire vector — are present and pass. The seam changes (`package(&Payload)`, `discriminate -> (Tag, Payload)`) are byte-faithful for the v0.1 entr path. All cross-kind length-binding directions are empirically rejected. The single deferred test (corrupted-mnem BCH correction) is a coverage gap, not a correctness defect — I proved the underlying path correct at all 5 mnem lengths with a throwaway probe — so it does not rise to Critical/Important. It is filed below as a Minor with a strong recommendation to land it this phase.

## Minor

- **[M1] `crates/ms-codec/tests/mnem.rs` — the corrupted-mnem-correction test (plan Task 1.3 Step 5b / plan-R0 I2) is MISSING, and the deferral justification recorded by the implementer is factually FALSE.** The plan mandated: "add a mnem round-trip-through-correction test (corrupt a mnem string by ≤4 symbols → corrects → decodes to the same `Payload::Mnem`)." Only the CLEAN-passthrough case (`mnem_decode_with_correction_clean_passes`, mnem.rs:42) was added. The argument that corrupted-mnem is "already tested in `bch_all_lengths.rs`" is wrong: `bch_all_lengths.rs:14` `LENGTHS=[16,20,24,28,32]` are entropy **byte** lengths, and `:32`/`:107` encode `Payload::Entr(e)` → its 1–4-error correction loop runs over entr **string** lengths {50,56,62,69,75} ONLY. The mnem string lengths {51,58,64,70,77} have NEVER been run through corrupted-input correction in the suite.
  - **Why this is only Minor (not Critical/Important):** I ran a throwaway probe corrupting mnem strings by k=1..4 symbols at all 5 lengths — every case corrects and decodes to the right `Payload::Mnem{language:2,..}` with the right entropy (output: 20/20 `OK`). The path is empirically correct. The reason it follows: post-v0.2.1-fix, the BCH layer sees only the codex32 symbol vector and verifies against ONE fixed target (`MS_REGULAR_CONST = 0x10ce0795c2fd1e62a`, `POLYMOD_INIT = 0x1`); `decode_regular_errors(residue, symbols.len())` is correctly parameterized by the symbol count, so the disjoint mnem symbol-lengths are handled by the same length-generic machinery. Payload kind is invisible below `decode`.
  - **Why it should still be landed THIS phase (not deferred):** `BUG_decode_with_correction_length_divergence.md` documents that this exact module shipped a length-divergent bug undetected for a full cycle precisely because of a 16-byte-monoculture fixture. The mnem lengths are a brand-new, previously-unexercised length set on the BCH-correction path. "Empirically correct today" is not "guarded against regression," and the plan made this test a named deliverable. Cost is ~12 lines. **Fix:** add a test that encodes `Payload::Mnem` for each of {16,20,24,28,32}-B entropy, corrupts ≤4 data-part symbols, asserts `decode_with_correction` recovers the same `Payload::Mnem` + correct entropy (the probe I wrote is a ready template). I am not gating GREEN on it because the path is proven correct, but it is the single most valuable test still owed and the per-phase regression-coverage discipline calls for it.

- **[M2] `crates/ms-codec/src/decode.rs:47` — pre-dispatch `allowed: VALID_STR_LENGTHS` is a partial answer (plan-R0 I1 residue).** The post-dispatch kind-bind (`:58-62`) correctly reports `allowed_for_kind(kind)`. The pre-dispatch literal at `:47` only ever fires when `len ∉ union` (neither set is "correct"), so reporting the entr set is defensible but arbitrary. Optional: report the union, or both sets. Not blocking — no real mnem-length input ever reaches this literal (the disjoint-set probe confirms mnem lengths pass the union gate and get the correct kind-appropriate `allowed` downstream).

- **[M3] `error.rs` alphabetical-placement is well-intentioned but underdetermined.** `MnemUnknownLanguage` is placed 2nd (after `Codex32`, before the unsorted pre-existing block); Display arm placed consistently. Since the pre-existing variants are intentionally NOT retro-sorted (per spec), "alphabetical placement" has no strict anchor — current placement (M after C) is reasonable and the Display string matches the plan verbatim (`"unknown mnem wordlist-language code: {0}"`). No action needed.
