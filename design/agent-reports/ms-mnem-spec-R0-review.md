# ms-mnem SPEC R0 review — wordlist-language hint

**Reviewer:** opus architect (R0, mandatory pre-plan/code gate)
**SPEC:** `design/SPEC_ms_mnem_wordlist_language.md` (DRAFT, Cycle 1 of the v0.2 split; base `master 4e5266a`)
**Prior R0 (both-features draft, RED 3C/4I):** `design/agent-reports/ms-v0-2-spec-R0-review.md`
**Primary sources verified against:** codex32 0.1.0 `/tmp/cx32_inspect/codex32-0.1.0/src/{lib.rs,field.rs}`; ms-codec `crates/ms-codec/src/{consts.rs,payload.rs,envelope.rs,decode.rs,encode.rs,error.rs,inspect.rs,lib.rs}`; ms-cli `crates/ms-cli/src/{language.rs,main.rs,format.rs,cmd/{encode,decode,inspect}.rs}`; `Cargo.toml`s; `.github/workflows/rust.yml`. **An empirical spike against the real codex32 0.1.0 crate confirmed the C1 fix end-to-end (construct + `data()` round-trip, all 5 lengths).**

## Verdict: RED (0C / 1I)

The spec's central de-risk — the **C1 byte-aligned re-frame** — is **fully sound and empirically proven**: all five entropy lengths construct through the existing `from_seed(&[u8])` path with no `IncompleteGroup` rejection, `data()` recovers the `[0x02][lang][entropy]` bytes exactly (no trailing-bit loss), and the string-length set is precisely `{51,58,64,70,77}` as the spec states. entr byte-identity is preserved (purely additive `0x02` arm). The wire design, error/const naming, AUTO routing, no-GUI-lockstep claim, CI reality, and SemVer are all correct.

**One Important finding blocks GREEN:** §3/§5 materially **under-scope the decode + inspect CLI/codec rework**. Three v0.1-hard-coded gates (`decode()` rule-9 length check, the ms-cli `inspect::analyze()` rule-walker's rule-8 prefix + rule-9 length pushes, and the ms-cli `decode::run()` `unreachable!()` non-Entr arm) will each reject or panic on a perfectly valid `0x02` mnem string. A plan author who lifts §3/§5 literally would ship an `ms inspect <mnem>` that prints `FAIL: would NOT decode` and an `ms decode <mnem>` that panics. The spec must enumerate these sites so the plan covers them. This is a scope-completeness gap, not a wire-format defect — hence Important, not Critical.

## Important

### I1 — §3/§5 under-scope the decode + inspect rework: three v0.1-hard-coded gates will reject/panic on a valid `0x02` mnem string
- **Evidence (three concrete sites, all on current `master 4e5266a`):**
  1. **`decode()` rule-9 length gate** — `decode.rs:29` rejects any `s.len() ∉ VALID_STR_LENGTHS` (= `{50,56,62,69,75}`, `consts.rs:33`) *before* parse. A mnem string is length `{51,58,64,70,77}` → `Error::UnexpectedStringLength` *before* `discriminate()` is ever reached. §3 says "Decode via `Parts::data()`" and §2 says the decoder "dispatches on the prefix byte first," but rule-9 fires first and the spec never says to widen the length set. **A separate `VALID_MNEM_STR_LENGTHS` const exists in §3 — but §3 does not say `decode()` must accept the *union* of entr+mnem lengths at rule 9.**
  2. **ms-cli `inspect::analyze()` rule-walker** (`crates/ms-cli/src/cmd/inspect.rs:46-92`) hard-codes the v0.1 rules: line 79 pushes `"non-zero-prefix"` for any `prefix_byte != 0x00`; line 83 pushes `"unexpected-string-length"` for any `str_len ∉ VALID_STR_LENGTHS`. A `0x02` mnem string (prefix `0x02`, length e.g. 51) therefore reports **`FAIL: would NOT decode v0.1`** with reasons `non-zero-prefix` + `unexpected-string-length` — the exact OPPOSITE of §5's "shows `kind: mnem` + the decoded language." The reason-text table (`inspect.rs:96-104`) and the hard-coded `[50, 56, 62, 69, 75]` literal at line 96 also need the mnem arm. (The ms-codec `inspect()` *codec* function at `inspect.rs:34-63` is already lenient and surfaces `prefix_byte=0x02` + payload cleanly — so the codec side is fine; the gap is entirely the ms-cli analyzer + the `InspectReport`/`InspectReportJson` lacking `kind`/`language` fields.)
  3. **ms-cli `decode::run()` payload match** (`crates/ms-cli/src/cmd/decode.rs:53-58`): `match payload { Payload::Entr(b) => …, _ => unreachable!("ms-codec v0.1 only decodes to Payload::Entr") }`. Once the codec returns `Payload::Mnem` for `0x02`, this `unreachable!()` **panics**. §5 says decode "emits the phrase in the on-wire language" but does not flag that this match arm + the language-selection logic (`decode.rs:44-48` currently always defaults English when no `--language`) must change to source the language from `Payload::Mnem.language`.
- **Why it matters:** §3 ("Codec changes") and §5 ("CLI surface") read as if mnem decode/inspect is a thin additive dispatch. It is not — it touches `decode()` rule-9, the ms-cli `analyze()` rule-walker (+ its reason table + the two `InspectReport*` structs gaining `kind`/`language`), and the `decode::run()` match + language-selection branch. A plan that doesn't enumerate these will leave `ms inspect <mnem>` broken and `ms decode <mnem>` panicking, and the §7 test plan's "inspect shows kind=mnem" / "decode recovers the language" cells would be the only thing catching it — late, at Phase 2, not at spec time.
- **Fix:** §3 and §5 enumerate, as explicit deliverables: (a) `decode()` rule-9 accepts the **union** entr∪mnem length set (or dispatches length-validity per discriminated kind); (b) ms-cli `inspect::analyze()` + `reason_text()` gain a `0x02` arm (prefix `0x02` + mnem-length set are *valid*, not failures), and `InspectReport`/`InspectReportJson`/`InspectJson` gain `kind` (+ decoded `language`) fields; (c) ms-cli `decode::run()` replaces the `_ => unreachable!()` arm with a `Payload::Mnem { language, entropy }` arm that sources the wordlist from the wire byte (and decides the `--language` override/precedence — see Minor M1). Note `DecodeJson` (`format.rs:48-54`) already HAS a `language` field; for mnem its *semantics* change (wire-sourced, `language_defaulted=false`) rather than the field being new — §5's "gains a `language` field" is imprecise for decode (true for inspect, semantic-only for decode); correct the wording.

## Minor

### M1 — decode `--language` precedence on a `0x02` mnem string is unspecified
- **Evidence:** `decode.rs:44-48` today: `Some(l) => use it; None => default English`. For a mnem string the wire *carries* the language. §5 says "no `--language` guess … when the wire carries it," but does not say what happens if a user *explicitly passes* `--language french` while decoding a `0x02`-`chinese-simplified` string.
- **Fix (state one in §5):** recommend the on-wire language is authoritative for `0x02`; an explicit `--language` that *conflicts* should either be ignored-with-stderr-note or error. (Cleanest: wire wins, explicit-conflict → stderr advisory. A silent override would re-introduce the §6.3 footgun the cycle exists to kill.) One sentence in §5 resolves this.

### M2 — `discriminate()` source-comment + `decode.rs` rule-7 path reference stale `0x01` / `RESERVED_TAG_TABLE` semantics
- **Evidence:** `envelope.rs:88-90` doc-comment promises "`0x01` → v0.2 entr-share path, `0x02..` → kind-specific dispatch" — the prior R0 C2 killed the `0x01`-share idea; this cycle should reword the comment to "`0x02` → mnem" and drop the `0x01`-share promise (it is K-of-N's, and that cycle will NOT use `0x01`). `error.rs:32/37/106` + `decode.rs:44` doc-comments say "RESERVED_TAG_TABLE" but no such const exists (the prior R0 I1; §10 footgun already flags `TAG_ENTR`+`RESERVED_NOT_EMITTED_V01` as the real names). These are comment-only and the spec §10 already captures the const-naming; just ensure the plan reworks the `envelope.rs` discriminate doc-comment when it touches that function (it must, for the `0x02` arm).
- **Fix:** plan notes the `discriminate()` doc-comment reword as part of the dispatch change; no spec edit strictly required (§10 covers the const names), but a one-line note in §3 that the `0x01`-share doc-comment is corrected to `0x02`-mnem avoids a reviewer flag at phase-1.

### M3 — `Payload::Mnem` decode must route through `Payload::validate()` (length check) — confirm it's wired, and that the high-nibble-reserved check lives in `discriminate()`
- **Evidence:** §3 says `entropy` is "validated ∈ {16,20,24,28,32} bytes exactly as `Entr`" and the language low-nibble/high-nibble checks happen in `discriminate()` (`data[1]` validation → `MnemUnknownLanguage`). The current `Payload::validate()` (`payload.rs:50-63`) only has an `Entr` arm. The plan must add a `Mnem` arm to `validate()` (mirror the `VALID_ENTR_LENGTHS` check) AND keep the language-byte validation in the envelope (since `Payload::Mnem` carries a `u8` language with no intrinsic table). §3 implies both but should state the split explicitly: **length → `Payload::validate()`; language-byte → `discriminate()`** (so a malformed language byte fails at the seam, not deep in the CLI). Minor because §3 covers the substance; just make the validation-site split unambiguous.

### M4 — pre-existing `enum Error` is NOT alphabetical; state the (non-)retro-sort decision
- **Evidence:** `error.rs:9-80` order is `Codex32, WrongHrp, ThresholdNotZero, ShareIndexNotSecret, TagInvalidAlphabet, UnknownTag, ReservedTagNotEmittedInV01, ReservedPrefixViolation, UnexpectedStringLength, PayloadLengthMismatch, TooManyErrors` — not alphabetical. §3/§10 say "new variants alphabetical (project convention)" + "State whether pre-existing variants are retro-sorted." The spec already self-flags this; the plan must pick: insert `MnemUnknownLanguage` alphabetically among existing variants (mixed order) OR retro-sort (churn). Mirror toolkit's `error-rs-retroactive-alphabetical-sort` deferral (do NOT retro-sort) to minimize diff. Minor — spec already raised it; just resolve at plan time.

## Notes
- **The C1 re-frame is the headline win and it is airtight.** The bit-aligned design died on `sanity_check` for 3/5 lengths; the byte-aligned design passes all 5 and — crucially — encodes through the *existing* `from_seed(&[u8])` and decodes through the *existing* `c.parts().data()` with zero new bit-path code. The spike (`/tmp/mnem_spike`, throwaway) constructed and exact-round-tripped all five against the real pinned codex32 0.1.0. The plan should still re-run this spike in-repo per §7/§9 (Phase 0 is load-bearing), but the format risk is retired.
- **I1 is a completeness gap, not a wire defect** — every site is a known v0.1 hard-coding that the spec's own §3/§5 narrative glosses. Folding it is mechanical: enumerate the three sites + the `InspectReport*` field additions + the `validate()`/`kind()` Mnem arms (M3) as explicit deliverables. Once §3/§5 list them, the §7 test plan's existing inspect/decode cells become real gates.
- **Re-dispatch this R0 after the fold** (per CLAUDE.md, the reviewer-loop continues after every fold; the I1 fold touches §3, §5, and §7 and is itself drift-prone — e.g. the decode-rule-9 union-length decision and the inspect `kind`/`language` JSON additions must stay consistent across §3/§5/§7).
- **Re-grep at plan time** — citations above are `master 4e5266a` snapshots per CLAUDE.md "citations decay"; the `0x01`-share + `RESERVED_TAG_TABLE` stale doc-comments (M2) are exactly the kind of decay to re-verify.
- Scope hygiene is good: zero K-of-N leakage, and the spec correctly inherits the prior R0's C1/C2/C3/I1 fixes (byte-align; drop `0x01`; ms-codec-no-CI; real const names). The split into mnem-first was the right call — mnem is a small, self-contained, byte-aligned win that unblocks the §6.3 hazard independent of the K-of-N crypto re-derivation.
