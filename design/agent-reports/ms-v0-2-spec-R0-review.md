# ms-v0.2 SPEC R0 review — K-of-N shares + mnem

**Reviewer:** opus architect (R0, mandatory pre-plan/code gate)
**SPEC:** `design/SPEC_ms_v0_2.md` (DRAFT, base `master 4e5266a`)
**Primary sources verified against:** codex32 0.1.0 `/tmp/cx32_inspect/codex32-0.1.0/src/{lib.rs,field.rs,checksum.rs}`; v0.1 SPEC `design/SPEC_ms_v0_1.md`; ms-codec `crates/ms-codec/src/{consts.rs,payload.rs,envelope.rs,decode.rs,encode.rs,error.rs,bch.rs,bch_decode.rs}`; ms-cli `crates/ms-cli/src/{language.rs,main.rs,cmd/derive.rs}`; `Cargo.toml`s; `.github/workflows/rust.yml`; `rust-toolchain.toml`.

## Verdict: RED (3C / 4I)

The K-of-N *math* is sound and the Phase-A byte-identity gate is achievable. But **two of the spec's load-bearing wire-format claims are crypto/format-unsound as written** (C1 mnem bit-packing rejected by codex32's own sanity-check for 3 of 5 lengths; C2 the `0x01` prefix-byte discriminator is fundamentally incompatible with Shamir sharing). A third Critical (C3) is the false "ms-codec is fmt-gated / has CI" premise the test plan and footguns rest on. These must be resolved before a plan is written — C1 and C2 change the wire format itself.

## Critical

### C1 — mnem payload `[0x02][lang:4][entropy:8N]` is REJECTED by codex32's `sanity_check()` for N ∈ {20,24,32} (15/18/24-word seeds)
- **Evidence:** codex32 `lib.rs:111-118` `sanity_check`:
  `incomplete_group = (parts.payload.len() * 5) % 8; if incomplete_group > 4 { return Err(IncompleteGroup) }`.
  This runs **unconditionally** inside both `from_unchecksummed_string` (`lib.rs:143`) and `from_string` (`lib.rs:172`) — i.e. on the SPEC §4 mnem *encode* path AND on every decode. `parts.payload.len()` is the **5-bit-symbol count** of the payload region.
- **The math** (mnem payload bit-length = 8 + 4 + 8N = 12 + 8N; symbols = ⌈(12+8N)/5⌉):

  | N (bytes) | words | payload bits | symbols | `(symbols·5)%8` | verdict |
  |---|---|---|---|---|---|
  | 16 | 12 | 140 | 28 | 4 | OK |
  | 20 | 15 | 172 | 35 | **7** | **IncompleteGroup → REJECTED** |
  | 24 | 18 | 204 | 41 | **5** | **REJECTED** |
  | 28 | 21 | 236 | 48 | 0 | OK |
  | 32 | 24 | 268 | 54 | **6** | **REJECTED** |

- **Why:** codex32 assumes the payload decodes to a whole number of *bytes*; its incomplete-group bound (≤4 trailing bits) only holds for byte-multiple payloads. A `12+8N`-bit payload is `1.5 + N` bytes — the extra 12 (non-byte-multiple) bits push the trailing partial-symbol group above 4 bits for 3 of the 5 lengths. The SPEC's recomputed length set `{50,57,63,70,76}` is *arithmetically* correct (verified: `22 + ⌈(8N+12)/5⌉`), but 3 of those strings **cannot be constructed** — `from_unchecksummed_string` errors before returning.
- **Why the SPEC's own test plan would NOT catch it:** §8's mnem test ("a hand-computed reference vector for **at least one** (language, entropy) pair") only mandates one vector; if the author picks 12-word (N=16) or 21-word (N=28) it passes while 3 lengths are broken. "Self-round-trip" can't fire because encode itself fails. This is exactly the monoculture-fixture trap that hid the v0.2.1 BCH length-variant bug (`BUG_decode_with_correction_length_divergence.md`).
- **Fix (one of):**
  (a) **Re-frame the payload as byte-aligned.** Spend a full byte on the language: `[0x02][lang:8][entropy:8N]` = 16 + 8N bits = whole bytes → encodable via the *existing* `from_seed(&[u8])` path (no hand symbol-assembly, no `data()` hazard) and `data()` round-trips cleanly. Length set becomes `{51,57,63,69,76}`-ish — recompute. High nibble of the language byte stays reserved-zero exactly as the SPEC already wants. This dissolves C1 entirely and most of the §4 "do not use data()" complexity. **Recommended.**
  (b) Keep 4-bit lang but pad the payload to a byte boundary with a documented reserved nibble so the *symbol count* lands at an incomplete-group ≤4 for all N — but you cannot freely choose: `(symbols·5)%8` is fixed by total bit count, so you must pad to a multiple of 8 bits anyway, which is just option (a) by another name.
  (c) Abandon hand-assembly and verify empirically: the plan MUST include a Phase-0 spike that calls `from_unchecksummed_string` for ALL FIVE N and asserts success, before any other mnem work. (This is the *minimum* even if (a) is taken.)

### C2 — the `0x01` prefix-byte discriminator is incompatible with K-of-N sharing; SPEC §3 `derive_share` "re-stamps the 0x01 prefix" is crypto-incoherent
- **Evidence:** the v0.1/v0.2 type byte lives **inside** the codex32 payload (`envelope.rs:147-167` `package()` prepends `RESERVED_PREFIX` to the data fed to `from_seed`; SPEC §2 table row `0x01 | [0x01][entropy:8N]`). Shamir/Lagrange in codex32 operates **per payload symbol** across shares (`lib.rs:290-293`: each result symbol `j` is a linear GF combination of `shares[i].payload[j]`). The secret-at-S is itself one polynomial point.
- **Consequence:** a distributed share at index `≠ S` is `interpolate_at(defining_set, idx)` — its payload symbol-0 is a *Lagrange combination*, NOT `0x01`. Only the secret share (and any defining share you happen to choose to start `0x01`) carries a meaningful first byte. Therefore:
  1. **`derive_share` cannot "re-stamp" the `0x01` prefix.** Overwriting symbol-0 of an interpolated share (i) breaks the short BCH checksum, and (ii) makes the share no longer a valid point on the polynomial → `combine_shares` would interpolate the **wrong secret**. (Verify the Lagrange contract: `lib.rs:272-294` — every symbol, including the prefix region, must be the true evaluation for recovery at `Fe::S` to return the original secret. bip_vector_3 confirms: shares D/E/F are pure interpolations with no post-hoc byte surgery.)
  2. **Grouping invariant #2 ("gate on the prefix byte BEFORE id") cannot be applied to distributed shares** — they have no stable prefix byte. The whole §2/§5 "dispatch on `0x01`" design only sees a real prefix on the secret-at-S share, which is *the one share you do NOT distribute* (SPEC §3 + §10 footgun: "the secret share at S is NOT distributed").
- **Root cause:** BIP-93 already has share discriminators — the **threshold char** (`0` vs `2..9`, `lib.rs:185-196`) and the **share-index** (`s` vs other, `lib.rs:198,202-204`). A codex32 share is identified by `threshold != 0` (equivalently index `!= s`), NOT by a payload byte. The prefix-byte scheme the v0.1 SPEC §5 reserved for "K-of-N entr" was designed before anyone traced that the byte is *inside* the shared region.
- **Fix:** drop `0x01` as a share discriminator. Distinguish shares by the codex32 **threshold field** (`k ∈ 2..9`) + share-index, exactly as BIP-93 does. The secret payload that gets shared should be `[0x00][entropy]` (the *same* bytes a v0.1 entr secret has) so that recovering the secret-at-S yields a clean v0.1-shaped `0x00` payload that the existing `decode`/`discriminate` path reads unchanged. Then:
  - `encode_shares(tag, Threshold(k), …)`: shares carry threshold-char `k`, indices ≠ `s`; their payloads are interpolation outputs (opaque). The secret-at-S (threshold `k`, index `s`) is NOT emitted.
  - decode/grouping dispatch: `threshold == 0` → v0.1/entr/mnem single-string path; `threshold ∈ 2..9` → share path, group by `id`. This replaces invariant #2's broken "prefix-byte gate" with a "threshold-field gate" — and it actually works for distributed shares.
  - Update SPEC §2 registry, §2 invariants #1/#2/#3 restatement, §3, and §5/§9 phasing accordingly. The `0x02` mnem discriminator is unaffected *as long as* C1 is fixed and mnem stays threshold-0 single-string (it is unshared in v0.2). But note: with shares now keyed on threshold-char, the mnem `0x02` byte is still only meaningful on a threshold-0 string — fine, mnem is never shared.
  - **This invalidates the SPEC's claim that the prefix-byte migration "reserved exactly this" (§1) — it reserved a byte that turns out to be unusable for sharing.** The plan must document the correction and amend `MIGRATION.md` (which carries the same four invariants) + the `envelope.rs` source comments that promise "`0x01` → v0.2 entr-share path."

### C3 — "ms-codec IS fmt-gated" / "ms-codec has its own separate workflow" are FALSE; ms-codec has no CI at all
- **Evidence:** the *only* workflow file in the repo is `.github/workflows/rust.yml` (`find … -path '*workflow*'` → one result). It is named `rust (ms-cli)` and its `paths:` filter is **`crates/ms-cli/**` only** — it never builds, tests, clippies, or fmt-checks `crates/ms-codec`. A repo-wide `grep -rn 'cargo fmt|fmt --check|rustfmt' .github/` returns **nothing**. The workflow comment claims "ms-codec has its own separate workflow" but no such file exists (`git log --oneline -- .github/` shows only two commits, both ms-cli Cycle-B PE).
- **Why it matters:** SPEC §8 ("ms-codec is fmt-gated, unlike the toolkit") and §10 footgun ("mnemonic-secret IS fmt-gated — run `cargo +stable fmt --check --all`") are load-bearing process claims that are simply wrong. A cycle that *relies* on CI to catch fmt/clippy/test regressions on the new ms-codec share + mnem code will ship unverified — there is no gate. (`rust-toolchain.toml` does pin `1.85.0` + `components=["rustfmt","clippy"]`, so the toolchain is available locally, but nothing in CI runs them on ms-codec.)
- **Fix:** correct the SPEC's two fmt/CI assertions, and have the plan EITHER (a) add an ms-codec CI workflow (build/test/clippy/fmt) as a Phase-E lockstep deliverable — strongly advisable given this cycle adds crypto to ms-codec — OR (b) explicitly state ms-codec verification is local-only (`cargo test -p ms-codec`, `cargo clippy -p ms-codec --all-targets -D warnings`, `cargo fmt --check`) and make that a hard manual gate at every phase. Do not write "CI gates it" when CI does not.

## Important

### I1 — SPEC cites a const `RESERVED_TAG_TABLE` that does not exist in the source
- **Evidence:** invariant #3 (§2 line 27) and §4 references name `RESERVED_TAG_TABLE` (`entr,seed,xprv,mnem,prvk`). The actual `consts.rs` has **no** such symbol: `entr` is `TAG_ENTR` (`consts.rs:36`) and the other four are `RESERVED_NOT_EMITTED_V01` (`consts.rs:39`). `error.rs:32/37/106` doc-comments reference "RESERVED_TAG_TABLE" conceptually but there is no const. The anti-collision invariant #3 says "refuse any `id` ∈ `RESERVED_TAG_TABLE`" — the plan needs a concrete predicate (`tag == TAG_ENTR || RESERVED_NOT_EMITTED_V01.contains(tag)`), and since this cycle EMITS `mnem`, `mnem` must move out of `RESERVED_NOT_EMITTED_V01` yet STAY in the anti-collision blocklist. The SPEC §4 says "Remove `mnem` from `RESERVED_NOT_EMITTED_V01`" but invariant #3's reserved list still includes `mnem` — reconcile: emit-allowed ≠ collision-allowed.
- **Fix:** plan introduces an explicit `RESERVED_ID_BLOCKLIST` (the full 5-tag set, used for anti-collision on random `id`) distinct from `RESERVED_NOT_EMITTED_V01` (decoder-reject set, from which `mnem` is removed). Grep-verify the real names per CLAUDE.md "citations decay" discipline.

### I2 — share-payload validity / `combine_shares` → `Payload::Entr` assumes the recovered secret is a clean entr, but the secret payload's prefix byte and length must be defined
- **Evidence:** SPEC §3 `combine_shares` = `interpolate_at(_, Fe::S)` → strip prefix → `Payload::Entr`. With C2's fix the secret-at-S payload is `[0x00][entropy]`, and `discriminate()` (`envelope.rs:91-137`) already strips `0x00` and length-validates via `Payload::validate()` (`payload.rs:50`). Good — but the SPEC currently says the secret carries `0x01`, and §3 routes the combine output to `Payload::Entr` without going through `decode`/`discriminate`'s rule-8 prefix check. Also: `interpolate_at` requires **all shares the same string length** (`lib.rs:238`); since entr length is fixed by N, this holds — but the plan must assert it and surface `MismatchedLength` as an ms error (the SPEC §3/§6 lists `ThresholdNotPassed/MismatchedId/RepeatedIndex/MismatchedThreshold` but **omits `MismatchedLength` and `MismatchedHrp`**, both reachable from `interpolate_at`, `lib.rs:239,242`).
- **Fix:** route `combine_shares` output through the existing `discriminate()`/`Payload::validate()` so prefix + length checks are not re-implemented; add `MismatchedLength` and `MismatchedHrp` to the surfaced-error list in §3/§6.

### I3 — `interpolate_at` `RepeatedIndex` is detected LAZILY (only during the Lagrange double-loop), and threshold-count check uses `shares.len()`, not distinct-index count
- **Evidence:** `lib.rs:230` checks `s0.threshold > shares.len()` (count, not distinct count); `RepeatedIndex` is only raised inside the `i,j` multiply loop at `lib.rs:283-285`. So passing k shares where two share an index could pass the count gate and only fail later (or, for the target-is-an-input short-circuit at `lib.rs:259`, return a wrong/duplicate share without the repeated-index ever being checked). SPEC §3(b) claims the procedure "guarantees unique indices + ≥threshold count" — that's the *generation* side, but `combine_shares` accepts arbitrary user input and must not trust it.
- **Fix:** `combine_shares` should pre-validate distinct indices + count ≥ k itself (don't rely on `interpolate_at`'s lazy/partial checks), then call `interpolate_at(_, Fe::S)`. Add a test: k shares with a duplicated index → clean `RepeatedIndex` (not a panic / wrong recovery).

### I4 — GUI `schema_mirror` + manual + toolkit-inspect lockstep is under-scoped for the wire change
- **Evidence:** SPEC §7 lists the lockstep chain. Per toolkit `CLAUDE.md`, the GUI `schema_mirror` gate is **flag-NAME parity only, NOT `--json` wire-shape**, and it is a *lagging* indicator (fires on the next GUI pin bump). This cycle adds two subcommands (`ms share`/`ms combine`) **and** changes the `--json` wire-shape of `ms decode`/`ms inspect` (new `language` field for mnem; new "share k-of-n" shape) — the latter has **no automated gate** and must be hand-coordinated. Also: the toolkit `inspect`/decode "learns to dispatch `0x02`/`0x01`" — but with C2's fix the toolkit must dispatch on the **threshold field** for shares, not `0x01`; and the toolkit consumes ms-codec as a **crates.io lib + ms-cli as a git tag** (memory: toolkit is NOT on crates.io, ms IS), so the re-pin is `ms-codec 0.2.1→0.3.0` (crates.io) and `ms-cli v0.5.1→v0.6.0` (tag). SPEC §7 has the versions right but the plan must make the wire-shape (non-gated) updates explicit in the same/paired PR.
- **Fix:** plan enumerates (a) GUI `ms.rs` schema flag-name additions for `share`/`combine` (gated), (b) GUI consumers of `ms decode --json`/`ms inspect --json` mnem/share fields (NOT gated — manual self-update per paired-PR rule), (c) toolkit inspect/decode share dispatch keyed on threshold not `0x01`, (d) manual `43-ms.md`. Note that `gui-schema` is reflective (`main.rs:110-114` `GuiSchema`), so the new subcommands auto-appear in the emitted schema — the hand-maintained mirror is what must catch up.

## Minor

### M1 — AUTO-default routing: `--hex` has no language so it must route to entr; confirm decode-time language for `0x00`/`0x01` is "unknown/record-it"
- §4/§5: English or `--hex` → `0x00` entr (byte-identical, Invariant-4 clean). Sound and matches `ms derive`'s "DEFAULT" annotation convention (`cmd/derive.rs:1-5`, language load-bearing). One nit: a user who encodes a *non-English* phrase but the AUTO logic keys on "is `--language` English-or-absent" — make sure "absent `--language` + non-English phrase" cannot silently emit `0x00` (it can't detect non-English from bytes). The §6.3 hazard is only fixed when the user explicitly passes `--language`. State that AUTO→mnem requires an explicit non-English `--language`; absent it, entr is emitted and the language hazard persists (as in v0.1). The SPEC implies this but should say it.

### M2 — output-class for `ms share` (§5/§64 self-flagged R0 question)
- A single share reveals nothing; the **set** of k+ shares is secret-equivalent. `ms share` emits the whole N-share set to stdout in one invocation → correctly classed `private key material (can spend)`. Confirmed sound. `ms combine` (emits secret entropy) and `ms decode`/`inspect` of entr/mnem → same class. Agree with the SPEC. (Minor: a single share emitted in isolation — not a v0.2 CLI path — would be watch-only-ish "reveals nothing alone"; not applicable since `ms share` always emits the set.)

### M3 — `Threshold::ZERO` vs codex32's `0` and the byte-identity wrapper
- §3/§9 Phase A: `encode` → `encode_shares(tag, Threshold::ZERO, &[p])`. For ZERO the wrapper must reproduce `package()` exactly: `from_seed(HRP, 0, tag, Fe::S, [0x00]||payload)` (`envelope.rs:161-167`). This IS achievable byte-for-byte (the inputs are identical; `from_seed` is deterministic). Phase A's SHA-vector gate is the right gate. Nit: `Threshold::new` rejects 1 and >9 (`InvalidThreshold`), but `Threshold::ZERO` must remain constructible as a distinct associated const (not via `new(0)` — keep `new` to `2..=9`, mirroring codex32's `from_seed` accepting 0 but `InvalidThresholdN` for 1). Confirm `ZERO` is a const, not `new(0)`.

### M4 — error-variant alphabetical ordering applies to `enum Error` AND its `Display` match
- `error.rs` `Error` is currently NOT alphabetical (Codex32, WrongHrp, ThresholdNotZero, …). Per CLAUDE.md the NEW variants (`InvalidThreshold`, `MnemUnknownLanguage`, the surfaced `Mismatched*`/`RepeatedIndex`/`ThresholdNotPassed`) must be inserted alphabetically *and* their `Display` arms ordered to match. SPEC §6 says "alphabetical per project convention" — good; the plan must also note the pre-existing non-alphabetical v0.1 variants are NOT retro-sorted in this cycle (mirror toolkit's `error-rs-retroactive-alphabetical-sort` deferral) to avoid churn, OR sort them — pick one and state it.

### M5 — `derive_share` public API is likely unnecessary once C2 is fixed
- With sharing keyed on threshold-char, `encode_shares` can derive ALL n distributed shares internally (build the k defining points, then `interpolate_at` for the remaining n−(k−1) indices) and return n strings directly. Exposing a separate `derive_share(group, target)` invites the "re-stamp prefix" footgun and the misuse of deriving the S share. Recommend `encode_shares` returns the full N-share set and `derive_share` is NOT public (or is a private helper). Re-evaluate the §3 API surface after C2.

## Crypto & wire verification (each ✓/✗)

- **Invariant-4 byte-identity feasibility** ✓ — `encode_shares(ENTR, ZERO, &[p])` reproducing `encode(ENTR,&p)` is byte-identical: both reduce to `from_seed(HRP,0,tag,Fe::S,[0x00]||payload)` (`envelope.rs:161-167`), deterministic. Phase-A SHA-vector gate is correct and the right ordering (before B/C/D). No byte shift. (M3 caveat on `Threshold::ZERO` being a const.)
- **K-of-N scheme correctness (secret@S, k−1 random defining shares, interpolate to derive, recover at S)** ✓ — matches BIP-93 + `interpolate_at` (`lib.rs:217-308`) + bip_vector_3 (`lib.rs:489-516`: defining set = `[S, A, C]` incl secret-at-S, derive D/E/F). The k DEFINING points INCLUDE the secret-at-S, so `payload_set.len()==k, [0]=secret, [1..k]=k−1` is correct.
- **`interpolate_at` precondition guarantees** ⚠ PARTIAL (I2/I3) — same length/HRP/threshold/id enforced eagerly (`lib.rs:238-252`); count check is `threshold ≤ shares.len()` (`lib.rs:230`, count not distinct-count); `RepeatedIndex` is LAZY (`lib.rs:283`). `combine_shares` must pre-validate distinct indices + count, and surface `MismatchedLength`/`MismatchedHrp` (omitted from SPEC §3/§6).
- **secret-share-at-S-not-distributed / combine interpolates at S** ✓ — sound; the S share is the recovery target, never handed out.
- **`encode_shares(tag, threshold, payload_set)` semantics** ⚠ — coherent for the DEFINING-set interpretation, BUT the `0x01`-prefix-on-every-share part is **✗ (C2)**. Re-shape: shares keyed on threshold-char; `encode_shares` should derive all N internally (M5); secret payload `[0x00][entropy]` not `[0x01]`.
- **k ∈ {2..9}, n ≤ 31** ✓ — `from_seed` map (`lib.rs:326-337`) accepts `{0,2..9}`, rejects 1 and >9 (`InvalidThresholdN`). 32 `Fe` values (`field.rs:160-191`) minus `S` = 31 distinct non-S indices → n ≤ 31. Both correct.
- **mnem bit-packing round-trippability** ✗ (C1) — REJECTED by `sanity_check` for N∈{20,24,32}. Recompute under byte-aligned `[0x02][lang:8][entropy]`.
- **mnem length set `{50,57,63,70,76}` = 22+⌈(8N+12)/5⌉** ✓ arithmetic / ✗ constructibility — formula verified exactly; all in the short bracket (totals 50–76 ∈ [48,93], pre-checksum lengths 37–63 < 81 so `from_unchecksummed_string` picks short, `lib.rs:123`); but 3 of the 5 are unconstructible (C1). Length set will change under C1's byte-aligned fix.
- **prefix dispatch `0x00`/`0x01`/`0x02`** ✗ for `0x01` (C2 — distributed shares have no stable prefix byte) / ⚠ for `0x02` (works only after C1; recoverable via `data()[0]` byte-aligned ONLY under the byte-aligned mnem layout — under the current 12+8N bit layout the prefix byte is NOT recoverable via `data()` because `data()` right-pads sub-byte tails and the SPEC itself forbids `data()` for mnem). The 16-word mnem=50 == 16-word entr=50 collision means dispatch MUST be on the prefix byte (correct instinct) — and under byte-aligned mnem, `data()[0]` is a clean `0x02`, recoverable BEFORE the bit-path, dissolving the §4 "slice symbols, don't use data()" hazard.
- **`id="entr"` for mnem unambiguous** ✓ conditionally — safe ONLY because dispatch is on the prefix byte (`0x02`) before `id`; with C2's threshold-keyed shares, the gate order is: threshold==0 → inspect prefix byte → `0x00`=entr / `0x02`=mnem (both `id="entr"`, distinguished by prefix); threshold∈2..9 → share. No collision. (The §49 alternative `id="mnem"` is unnecessary and would force a blocklist carve-out — keep `id="entr"`.)

## Notes
- The cycle is **two largely independent features** (K-of-N shares; mnem language hint) bundled because both "ride the prefix-byte discriminator." C2 demolishes that shared rationale for *shares* (shares must use the threshold field, not the prefix byte). mnem can still use `0x02`. After the C1/C2 fixes the two features no longer share a load-bearing mechanism — reconsider whether they should be **two cycles** (mnem is a small, self-contained, byte-aligned win that unblocks the §6.3 hazard; K-of-N is a larger crypto-wrapping effort whose wire design needs re-derivation). At minimum the A→E phasing should make mnem (now trivial, byte-aligned, via `from_seed`) independent of the shares re-design, so a C2 stall doesn't block the §6.3 fix. Recommend: keep one cycle only if both fixes land cleanly at plan-R0; otherwise split (mnem first).
