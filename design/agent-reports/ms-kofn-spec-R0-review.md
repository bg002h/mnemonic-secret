## Minor

### m1 — `Payload::wire_bytes()` is referenced as if it exists; it does not. The prefix-byte assembly lives only in `envelope::package()`.
SPEC §1:18, §2:51, §2:57 write `secret.wire_bytes()` = `[0x00]||entropy` / `[0x02][lang]||entropy`. There is **no `wire_bytes()` method on `Payload`** (`payload.rs:59-108` has only `validate`, `kind`, `as_bytes`). The actual `[prefix]||…` assembly is inline in `envelope::package()` (`envelope.rs:167-183`, `pub(crate)`), which takes `(Tag, &Payload)` — NOT a byte method on `Payload`. The design review's I3 specified the construction as `from_seed(HRP,k,id,Fe::S,[prefix]||payload)` (the package() shape), not a `wire_bytes()` abstraction — the SPEC *introduced* this method name. Implementable two ways: (a) add `pub(crate) fn wire_bytes(&self) -> Zeroizing<Vec<u8>>` on `Payload` and refactor `package()` to call it (cleaner; lets `encode_shares` reuse it for the secret-at-S), or (b) keep the assembly in the envelope and have `encode_shares` live in `envelope.rs` calling the same inline logic. **Fix:** the plan must name this — `wire_bytes()` is net-new code, not an existing API. (Also §1:51 / §2 step 1 write `tag.id()`; `Tag` has `as_str()`/`as_bytes()`, no `id()` — informal shorthand, fold to `tag.as_str()`.)

### m2 — the inspect rule-walker `analyze()` (`cmd/inspect.rs:49`) is the concrete site the share-first-class change must touch; SPEC §3 states the OUTPUT but never names `analyze()`.
A distributed share fed to `ms inspect` today trips **multiple** FAIL reasons at once: `threshold-not-zero` (`inspect.rs:58`, threshold∈2..9 ≠ 0), `share-index-not-secret` (`:62`, index≠'s'), almost certainly `non-zero-prefix` (`:83`, the share's interpolated garbage `data()[0]` ⇒ `kind==Unknown`), possibly `payload-length-mismatch`. SPEC §3 says inspect-of-share must be a first-class read (`kind: share`, threshold, id, index, `would_combine`), NOT `FAIL: would NOT decode`. That requires a **share branch in `analyze()` (and `emit_text`/`emit_json`) ahead of the §4 rule-2..10 walk**, gated on `threshold∈2..9`. Mitigating: `InspectReport` already carries `threshold`, `tag` (the id), `share_index`, `kind` (`inspect.rs:36-52`), so the fields exist — only the branch is new. **Fix:** §3/§8 should pin the inspect change to `analyze()`/`emit_*` so the plan task is concrete (the implementer would find it, but the SPEC's scope-enumeration goal is to pre-name it).

### m3 — toolkit consume-path: the new `ms_codec::Error` variants fall through TWO wildcard arms (`_ => 1` exit-code; `_ => "unhandled…"` friendly), which compile but mis-surface.
SPEC §4 says the toolkit "propagates `IsShareNotSingleString` via `?` → surfaced with a friendly map (add to `mnemonic-toolkit/src/friendly.rs` if not covered)." Verified: `ms_codec::Error` is `#[non_exhaustive]`, so adding variants won't break the toolkit build — but (a) `ms_codec_exit_code` (`error.rs:352`) has `_ => 1` (`:364`), so `IsShareNotSingleString`/`SecretShareSuppliedToCombine` silently map to exit **1**, which may be the wrong class for a usage/routing error (cf. ms-cli routes `ThresholdNotZero`→2; a "you pasted a share into a single-string reader" error arguably wants 64/usage or at least a deliberate choice); and (b) `friendly::friendly_ms_codec` (`friendly.rs:42`) has `_ => "unhandled ms_codec::Error variant: {:?}"` (`friendly.rs` last arm), so a share hitting toolkit `inspect`/`decode`/`repair` surfaces with that ugly debug string. The SPEC's "if not covered" understates it: they are definitively **not** covered. **Fix:** P3 must add explicit `friendly_ms_codec` arms + a deliberate `ms_codec_exit_code` arm for the new variants (esp. `IsShareNotSingleString`, the only one a toolkit user reaches). (Note `friendly_ms_codec` `error.rs:729`; the SPEC's `src/friendly.rs` path is correct.)

**Crypto re-verified by independent spike (adjudication #1, #3, #5):**
- **#1 construction soundness** — SPEC §2's construction (secret-at-S via `from_seed`; k−1 random defining shares at distinct non-`s` indices with same-byte-length getrandom payloads; `interpolate_at` for the remaining n−(k−1); return the n distributed shares; secret-at-S NOT returned) is the CORRECT codex32 procedure and matches `bip_vector_3` structure (defining set = `[S, A, C]` incl secret-at-S; D/E/F pure interpolations — `lib.rs:489-516`). Spike: recovery from a random k-subset of distributed shares reproduces the exact secret for entr ALL 5 lengths × k∈2..9, and mnem 5 lengths × {k=2,5,9}. **Returning the (k−1 defining + n−(k−1) interpolated) and excluding secret-at-S is correct** — the secret-at-S is the recovery target only; distributing it would hand out a "share" that combines trivially. **Subtlety on random defining shares: they MUST carry the same threshold/id AND same byte-length payload** — codex32 enforces same-string-length eagerly (`lib.rs:238`) and same-threshold/id (`:244/:250`); a length mismatch among the basis points would error. The SPEC's "CSPRNG payload of the same byte length" (§2 step 2) gets this right.
- **#3 byte-identity + ZERO id** — `encode_shares(tag,ZERO,1,&p) ≡ encode(tag,&p)` holds: both reduce to `from_seed(HRP,0,tag,Fe::S,[prefix]||payload)`, deterministic (spike: identical output across calls; `ms10entrsq…`). **The ZERO/n=1 path keeps id=tag (NOT random) — the SPEC IS explicit:** §2 ZERO bullet says "byte-identical to `encode(tag, secret)`… both reduce to `from_seed(HRP, 0, tag.id(), …)`" — `tag.id()` = the tag, not a random id (the random-id roll is gated under the `k∈2..=9` bullet only). No off-by-one (n=1, no defining shares, no interpolation). Confirmed sound. (The `tag.id()` shorthand → `tag.as_str()` is m1.)
- **#5 entr+mnem feasibility** — mnem sharing works (design-review-confirmed, spike-reconfirmed); secret `wire_bytes()` = `[0x02][lang]||entropy` is correct (matches `envelope.rs:175-182`, `consts.rs:43` VALID_MNEM_STR_LENGTHS); language byte (`wire[1]`) survives split/recombine for all lengths/languages. Correct.

**Adjudication #4 (scope completeness) — no missed v0.1-hardcoded gate that blocks the design:**
- Length-gate (`decode.rs:44` `is_known_length`): a distributed share has the SAME string length as the corresponding single-string (interpolate_at preserves payload byte-length; secret-at-S payload = `[prefix]||payload`). Computed: entr-share lengths {50,56,62,69,75} = `VALID_STR_LENGTHS`; mnem-share lengths {51,58,64,70,77} = `VALID_MNEM_STR_LENGTHS`. So a share PASSES the union length-gate → reaches parse → reaches `discriminate()`/threshold-check (good — the dispatch is reachable; SPEC §1 "No length collision… disambiguated by the threshold char, not length" is correct).
- `discriminate()` threshold check (`envelope.rs:105-109`) currently rejects ALL non-`'0'` thresholds with `ThresholdNotZero` — this is the exact site §2 must relax to route threshold∈2..9 → `IsShareNotSingleString`. SPEC covers it.
- The two un-named impl sites (`analyze()` rule-walker m2; toolkit wildcards m3) are Minor scope-completeness, not design-blocking.
