# Phase 3 (faithful) R0 design review — ms mnem — round 0

**Verdict:** RED (2C / 5I)

**Base SHA verified:** toolkit `e572888`, ms-codec/ms-cli on branch `ms-v0.2-kofn-mnem` at `7820049` (ms-codec 0.3.0 + ms-cli 0.6.0 already committed there, NOT yet on master/crates.io). All Phase-3 source citations re-grepped against these SHAs.

## Critical

- **[C1] Multi-slot bundles can carry DIFFERENT wire languages, but `effective_language` is "computed ONCE per run" — `verify_bundle.rs:2286-2419` and `wallet_import/overlay.rs:62-166` both derive per-cosigner with a SINGLE `language` parameter.** This is the sharp edge point 1 warned about, and the plan gets it wrong. Concrete proof:
  - `verify_bundle.rs::emit_full_path_parent_fingerprint_check` takes `language: CliLanguage` (one value, sourced as `args.language.unwrap_or_default()` at `:368/395/442/457/545/560`). Inside it, the loop `for (i, card) in &mk_cards` (≈`:2370`) decodes **each cosigner's own `ms1_str`** at `:2400` then re-derives at `:2411` `from_entropy_in(language.into(), &entropy)` — using the *single* function-scope `language` for *every* cosigner. A 2-of-3 bundle whose cosigner[0] ms1 is a Japanese `mnem` card and cosigner[1] is a Czech `mnem` card cannot both be re-derived correctly from one `language`. The plan's "computed once at seed intake and threaded through the pipeline" (plan:131) is structurally incoherent for this site.
  - `wallet_import/overlay.rs` has the identical shape: `language: CliLanguage` single param (`:62`), per-cosigner loop, decode each cosigner ms1 (`:127`), derive `from_entropy_in(language.into(), …)` (`:161`). Same bug.
  - **Why it blocks:** the plan's central abstraction (`effective_language` = one per-run value) is wrong for the two multi-cosigner derive sites. Threading a single value through will silently mis-derive (and thus *wrongly fail* the parent-fp / xpub-match cross-checks) any multisig bundle with ≥2 differently-languaged mnem cosigners — exactly the "faithful derive" the user asked for, broken at the multisig case.
  - **Fix:** make `effective_language` **per-card, not per-run**. At each consume site that has a `Payload` in hand, derive that card's language locally: `Payload::Mnem{language,..}` → `wire_code_to_bip39(language)`; `Payload::Entr(..)` → fall back to `--language`. For the two per-cosigner loops (`verify_bundle.rs:2400`, `overlay.rs:127`), compute the bip39 language *inside the loop from that cosigner's decoded payload*, not from the function parameter. The single-source paths (seed_intake, silent_payment, convert-from-phrase) keep a per-run value because they have exactly one source. Re-scope the plan's "computed once per run" to "computed per seed source; for multi-source bundles, per-card." Add a multi-language multisig test (cosigner[0]=ja mnem, cosigner[1]=cs mnem) to `verify-bundle` and `import-wallet`.

- **[C2] The canonical NAME map (I-NEW-1) cannot be a literal string-equality between `MNEM_LANGUAGE_NAMES` and the toolkit's `human_name()` — the two name vocabularies DIFFER for Chinese.** Verified ground truth:
  - `ms_codec::MNEM_LANGUAGE_NAMES` (consts.rs:47-58): `["english","japanese","korean","spanish","chinese-simplified","chinese-traditional","french","italian","czech","portuguese"]`.
  - toolkit `CliLanguage::human_name()` (language.rs:29-30): `"simplified-chinese"` / `"traditional-chinese"` — **word order reversed** vs the wire's `"chinese-simplified"` / `"chinese-traditional"`.
  - The plan (plan:139) says map "via an explicit match keyed on the canonical NAME (NOT a discriminant cast)" and the unit test asserts `wire_code_to_bip39(c)` agrees with `MNEM_LANGUAGE_NAMES[c]`. But if the implementer naively bridges `CliLanguage ↔ wire` by comparing `human_name() == MNEM_LANGUAGE_NAMES[code]`, the two Chinese variants will **never match** and either fail to map or (worse) map to the wrong/None language. The `as u8` divergence the plan flags is real and correct (toolkit Japanese=6 vs wire japanese=1, etc. — fully confirmed: toolkit order `English,SimplifiedChinese,TraditionalChinese,Czech,French,Italian,Japanese,Korean,Portuguese,Spanish` vs wire order above), but the proposed *fix* (name-keyed) is itself under-specified because the names aren't a shared vocabulary.
  - **Why it blocks:** "map through the canonical NAME" as written is a latent silent-miscompare for 2 of 10 languages — and Chinese is a high-value non-English case for this exact footgun. A name-keyed map that doesn't account for the `simplified-chinese`↔`chinese-simplified` swap is a Chinese-specific wrong-derivation.
  - **Fix:** define both directions as **explicit `CliLanguage ↔ u8` match tables** (not via string round-trip), e.g. `cli_language_to_wire_code(CliLanguage::SimplifiedChinese) => 4`, `wire_code_to_cli(4) => CliLanguage::SimplifiedChinese`, and separately `wire_code_to_bip39(code)` as an explicit `code → bip39::Language` match. The unit test should assert the *bip39::Language* identity (`wire_code_to_bip39(c) == CliLanguage::from(wire_code_to_cli(c)).into()`) and that `MNEM_LANGUAGE_NAMES[c]` is merely the human label — NOT that two name strings are byte-equal. Keep one assertion pinning `MNEM_LANGUAGE_NAMES[c]` to the *expected* label per code so the map can't silently re-order, but do the mapping by code, not by string-compare against `human_name()`.

## Important

- **[I1] `cmd/silent_payment.rs:113-119` (`to_master`) hardcodes `bip39::Language::English` and is NOT in the plan's Step-4 consume-site derive list as a language-honoring site.** The plan lists `silent_payment.rs:135/137` (the decode) but the actual derivation happens in the `to_master` closure at `:114`: `Mnemonic::from_entropy_in(bip39::Language::English, entropy)` → `derive_master_seed` → `Xpriv::new_master`. A non-English `mnem` ms1 fed to `mnemonic silent-payment` will decode the entropy fine but re-derive the seed under English → **wrong silent-payment address**, silently. This is a DERIVE site that mis-derives. The plan's Step 4 must explicitly route the ms1-branch (`:134-144`) through the wire language: decode → if `Payload::Mnem{language,..}` pass `wire_code_to_bip39(language)` into `to_master` (parameterize `to_master` on the language). Note the bare-phrase branch at `:148` also hardcodes English — that's acceptable (a bare phrase positional has no wire language and silent_payment has no `--language` on that path; confirm), but the ms1 branch is the gap.

- **[I2] `repair.rs:820` / `:892` `decode_with_correction` returns a `Payload` whose `Mnem` arm is unhandled, and the plan's advisory-suppression claim conflicts with the repair-derive flow.** Two sub-issues: (a) Step 4 lists `repair.rs:820/892` for the consume sweep but doesn't say what repair *does* with the recovered payload — confirm whether repair re-derives a seed (needs wire language) or only round-trips the corrected string (no derive). From the call sites it short-circuits with the corrected card; if it ever surfaces entropy for derivation downstream, that downstream needs the wire language. (b) `repair --md1`-style and the ms1 auto-repair path feed `seed_intake.rs:182` and `verify_bundle.rs:1258` — those re-enter the consume path, so the per-card language rule from C1 must hold there too. The plan should state explicitly that repair is round-trip-only (language-agnostic) OR route its output through the same per-card language. Add a "corrupt a Japanese mnem card by ≤4 symbols → repair → re-derives the Japanese seed" test.

- **[I3] The §6.3 advisory suppression is keyed on the wrong predicate at `bundle.rs:737` and `convert.rs:1017` — both use `args.language.unwrap_or_default()`, NOT `effective_language`.** Verified: `bundle.rs:737-742` fires the advisory iff `non_english_seed_advisory(args.language…, "an ms1 card")` and `bundle.any_secret_bearing()`. `convert.rs:1017-1023` fires iff target contains `Ms1` and `args.language…` is non-English. The plan (Step 6) says "suppress the advisory when the toolkit emits `mnem`" — but the *correct* gate is "suppress iff the emitted ms1 is a self-describing `mnem` card." Since the emit rule (Step 5) emits `mnem` for any non-English effective language, and the advisory ALSO fires for non-English, the two are *almost* coextensive — BUT only if both read the SAME language source. If the advisory keeps reading `args.language` while emit reads `effective_language`, a mnem-ja *source* with no `--language` flag (so `args.language` defaults English) will: emit a mnem-ja card (correct, language preserved) AND **suppress nothing because the advisory already saw English and didn't fire** — accidentally OK; but the inverse (English `args.language` defaulted, wire=ja source → still emits mnem) means the advisory's "record your language" never fires even though for an *entr* fallback it should. The clean fix: drive BOTH the emit kind AND the advisory from the same `effective_language`, and suppress the advisory exactly when `effective_language != English` *at an ms1-emit site* (because that site now emits mnem). Keep the advisory at slip39 (`slip39.rs:443`, `:658`) and at `convert`'s `Entropy`-target site (`convert.rs:1009` — raw-entropy target is genuinely language-losing). Re-state Step 6 in terms of `effective_language`, not `args.language`.

- **[I4] Step 4's consume-site list is missing the `verify_bundle.rs:644-664` descriptor-mode verify path** referenced by the comment at `verify_bundle.rs:2408-2409` ("Mirrors descriptor-mode verify path at `verify_bundle.rs:644-664`"). The plan lists `1227/1620/2400` but that comment points to a fourth derive site. Re-grep `from_entropy_in` across all of `verify_bundle.rs` (and `derive_slot.rs:78`, `:139`, which are the shared `derive_bip32_from_entropy` helpers that take a `language` param) and confirm every `from_entropy_in(language…, entropy)` reachable from an ms1 source is either (a) per-card-languaged per C1, or (b) explicitly out of scope with justification. `derive_slot.rs:78/139` are the production derive helpers — if they take `language: CliLanguage` and are called with the per-run value from a multi-language bundle, they inherit C1.

- **[I5] The dev-time `[patch.crates-io] ms-codec = { path = … }` will rewrite `Cargo.lock` (path source + drop the registry checksum), and the plan's "regenerated cleanly at ship" needs an explicit lock-diff gate.** Confirmed the toolkit already has a `[patch.crates-io]` block (Cargo.toml:16-17, currently only `miniscript`), so adding an ms-codec path override is mechanically fine. But: during dev, `cargo build` records `ms-codec` in `Cargo.lock` as `source = path+file://…` with no `checksum`. At ship, removing the override + bumping the pin to `"0.3.0"` requires a *fresh* `cargo build`/`cargo update -p ms-codec` so the lock re-acquires the crates.io registry source + checksum. The plan (Step 9) says "REMOVE the override → `cargo build`" — add an explicit assertion that the post-ship `Cargo.lock` contains `source = "registry+…"` and a `checksum` for ms-codec 0.3.0 (grep the lock), and that `cargo metadata --locked` passes (the existing CI guard from the v0.37.x cycle). Also: the publish must land ms-codec 0.3.0 on crates.io *before* the toolkit `cargo build` that regenerates the lock, or the build fails to resolve `"0.3.0"`. Sequence in Step 9 is right (publish → remove override → build) but the lock-source assertion is missing. Minor sub-note: caret `"0.3.0"` (plan:138) is correct — not exact.

## Minor

The English emit path is genuinely byte-identical *because* `Payload::Entr(entropy)` is unchanged and ms-codec's `package` builds `[0x00] ++ entropy` for Entr (SPEC §2; the entr byte-identity gate is Phase 1 Task 1.4). Confirm the toolkit test pins an actual golden ms1 string (e.g., a known English 24-word phrase → fixed ms1) rather than a self-referential round-trip, mirroring the codec's hand-computed vector discipline. The risk is low (Entr path untouched) but the assertion should be a pinned literal.

- **[M2] Scope adjudication (point 2): emitting `mnem` for a non-English *phrase* source (not just a mnem-ms1 source) is correct and in-scope.** The user chose "faithful derive + preserve." A non-English phrase + `--language ja` → mnem-ja card is the *same footgun closure* as preserving a mnem source: the resulting card is self-describing, so it does NOT regress and does NOT over-reach. Recommend KEEPING the plan's "any non-English effective language → mnem" rather than narrowing to "mnem-ms1 source only" — narrowing would leave the phrase-source footgun open (the exact thing v0.37.11's advisory was a band-aid for). Confirmed sound; no change needed, but flag for the record since the prompt asked.

## Confirmations (design points verified sound)

- **Emit-rule back-compat (point 2):** English → `Payload::Entr(entropy)` is byte-identical — confirmed the four current synthesize emit sites (`synthesize.rs:150/293/443/784`), `convert.rs:1217`, and `bundle`'s ms1 emit all build `Payload::Entr` today, and ms-codec's Entr `package` path is unchanged. The English common case does not regress.
- **`ms_codec::encode(tag, &Payload)` signature** (encode.rs:16) matches the plan's emit examples. The plan's `code:` field-name in some examples (plan:133 `Mnem { code, entropy }`) is a typo — the field is `language` (payload.rs:53); fold as a citation fix.
- **`Payload::as_bytes()` returns bare entropy for BOTH `Entr` and `Mnem`** (payload.rs:102-107) — so "extract entropy works for both kinds via `payload.as_bytes()`" (Step 4) is correct; the entropy-extraction sites that don't derive (e.g. pure `--import-json` entropy storage at `bundle.rs:1618`) only need the wildcard-reject arm relaxed, no language needed.
- **The wildcard-reject arms that must gain a `Mnem` arm** are confirmed at: `bundle.rs:1620` (`_ => Err`), `convert.rs:1454` (`_ => Err`), `silent_payment.rs:138` (`_ => Err`), `overlay.rs:130` (`Ok(_) => Err`), `verify_bundle.rs:2405` (`Ok(_) | Err(_) => continue`). All currently reject non-Entr. Plan Step 4's "replace every `_ => Err("not entropy")` arm" is accurate.
- **Wire-wins precedence** mirrors ms-cli decode's R0-M1 (SPEC §5, line 48): for a mnem source, wire language wins + stderr `note:`; conflicting `--language` warned-and-ignored. Sound and consistent with the sibling.
- **convert has NO ms1→ms1 re-emit path** (the Ms1-source arm's target match has a `_ =>` catch-all, no Ms1 target), so convert's ms1 emit (`:1217`) only fires from Phrase/Entropy/Seedqr sources where the language == `args.language` — the single-per-run value is correct *there*. (Contrast with C1's multi-cosigner sites.)
- **`MnemUnknownLanguage` validation** is enforced in `discriminate`/`validate` (payload.rs:76-79, `language >= 10 → Err`), so the toolkit's defensive `code ≥ 10` reject in `wire_code_to_bip39` (plan:139) is belt-and-suspenders but correct — never `unwrap` a bad index into `MNEM_LANGUAGE_NAMES`.
