## Important

- **[I6 — NEW] C1-RESIDUAL's dataflow prose is wrong for the import-json path, and the wrongness will misdirect the implementer.** The plan says (Step 4 bullet 5 + the "carrier" paragraph): "the derive at `bundle.rs:626` and the emit at `synthesize.rs:291/778` then READ `slot.language`." Verified against source, **the import-json flow never executes `bundle.rs:626`**:
  - `bundle.rs:626` lives in `resolve_slots` (`bundle.rs:448-709`), the `--slot @N.entropy=` arm. It derives from **raw hex** (`--slot @N.entropy=<hex>`), which carries NO wire language — so `slot.language` is *always* `None` there. There is no `ms_codec::decode` anywhere in `resolve_slots`.
  - The import-json path is `bundle_run_from_import_json` (`bundle.rs:1548-…`). It builds slots via `envelope_to_resolved_slots` (`wallet_import/json_envelope.rs:302-330`), where the **xpub comes from the mk1-chunk decode** (`mk1_card_to_resolved_slot`, `json_envelope.rs:364`), NOT from re-deriving the ms1 entropy. At `bundle.rs:1626` it merely **stuffs `entropy` onto the already-built slot** — no re-derive, no entropy↔xpub cross-check. The emit is `synthesize_descriptor` (`synthesize.rs:291`), not `synthesize_unified` (`:778`).
  - Consequence: round-1's claimed import-json bug "(a) re-derive cosigner i's xpub under English at `bundle.rs:626` → wrong xpub / wrong parent-fp cross-check" **does not occur** — there is no re-derive on this flow. The only real import-json bug is "(b) re-emit loses language," which the carrier→`synthesize.rs:291` emit fix correctly addresses. The two flows (`resolve_slots`→`synthesize_unified` at `:778`, and `bundle_run_from_import_json`→`synthesize_descriptor` at `:291`) are disjoint and the plan conflates them under one sentence.
  - **Fix:** rewrite the carrier paragraph + Step 4 bullet 5 to separate the two flows: **(import-json)** populate `slot.language` at `bundle.rs:1626` from the decoded payload; READ it ONLY at the emit `synthesize.rs:291` (`synthesize_descriptor`) — state explicitly there is NO re-derive on this flow, so the xpub correctness depends on the source envelope, not on the run `--language` (i.e. the import-json language bug is emit-only, not derive). **(`--slot @N.entropy=`)** `bundle.rs:619/626` + `bundle.rs:1369` derive from raw hex → `language: None` always (no wire language on a hex slot); set the new field to `None` (compile-forced). Drop the implication that `:626` reads a non-`None` `slot.language`.

- **[I7 — NEW] The `Payload::Entr`-only destructuring matches must be widened to BIND `Mnem`, not just relaxed to not-error — and the plan misses the silently-skipping one at `verify_bundle.rs:2403`.** The plan names three sites to "relax the `_ => Err` arm to accept `Mnem`" (`convert.rs:1454`, `bundle.rs:1620`, and implicitly silent_payment `:138`). But the verify_bundle per-cosigner loop's match is different in kind: at `verify_bundle.rs:2400-2404` it is `Ok((_tag, ms_codec::Payload::Entr(bytes))) => bytes` with `Ok(_) | Err(_) => continue` — a **`Payload::Mnem` card currently silently `continue`s, skipping the cross-check entirely** (not an error, a *silent verification gap*). The plan's Step 4 bullet 3 says only "resolve language PER COSIGNER inside the loop from each cosigner's decoded payload" — but the per-cosigner language resolution has nothing to act on while the `Mnem` payload is swallowed by `Ok(_) => continue`. **Fix:** Step 4 bullet 3 must explicitly say to widen the `verify_bundle.rs:2403` match to bind BOTH `Entr(bytes)` and `Mnem { language, entropy }` (extracting entropy + the per-card language), keeping the `Err(_) => continue` skip-on-undecodable. Add to the test list: a multisig `verify-bundle` whose `ms1[i]` is a `mnem` card must run the parent-fp cross-check (today it silently passes by skipping).
