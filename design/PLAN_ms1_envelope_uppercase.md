# PLAN — ms-codec 0.4.2: accept all-uppercase ms1 per BIP-173 (envelope/wire-layer canonicalization)

**Cycle:** ms-codec 0.4.1 → 0.4.2 (PATCH) + ms-cli exact-pin bump (`=0.4.1` → `=0.4.2`; ms-cli's own version unchanged unless its surface changes — the I9 precedent) · **Source SHA:** `952bebd` (master) · **Resolves:** `design/FOLLOWUPS.md::ms1-envelope-uppercase-bip173` (filed by toolkit v0.53.3 / audit M11).

## Problem (verified)

codex32 accepts consistent-uppercase strings (BIP-173/93: uppercase = the QR alphanumeric form; the checksum engine case-folds; `set_check_case` rejects only MIXED) — but ms-codec's wire layer then reads RAW string bytes: `discriminate` (envelope.rs:96-97 `c.to_string()` → `extract_wire_fields`) compares `fields.hrp != "ms"` (:100) and `share_index_byte != b's'` (:112-ish) case-sensitively, so a valid all-uppercase MS1 card fails `WrongHrp { got: "MS" }`. The same raw-byte pattern exists at the OTHER extraction call sites: `inspect` (inspect.rs:65-66) and `combine_shares` (shares.rs:200-201 threshold/share-index grouping; NB :302 is a test helper — R0-r1 M1), and any payload/sliced-byte reads off the raw string.

## Design — canonicalize the OWNED wire copy once per entry point

- New `pub(crate)` helper in envelope.rs (used from inspect.rs + shares.rs — R0-r1 M6): `fn wire_string(c: &Codex32String) -> String { c.to_string().to_ascii_lowercase() }` with a doc comment stating the soundness argument: **codex32 has already enforced consistent case + a valid checksum, so lowercasing the owned copy is canonical-form normalization of the BIP-173 uppercase (QR) form — NOT case-laundering (mixed-case never reaches here).**
- Use it at the THREE production extraction sites (complete census, R0-r1 M1 — shares.rs:302 is a TEST helper, leave it): `discriminate` (envelope.rs:96), `inspect` (inspect.rs:65), `combine_shares`' per-share extraction (shares.rs:200 — NOTE: subsumed by the C1(a) canonical vector below, which is the LOAD-BEARING mechanism for combine; keep the wire_string there as harmless defense-in-depth or drop it — R0-r3 M1-r3). Payload byte extraction riding the same copies is covered automatically; `envelope.rs:65`'s separator-less WrongHrp arm is covered once its input is the lowered copy.
- **COMBINE-side canonicalization (R0-r1 C1, option (a)):** the wire_string fix alone CANNOT make a mixed-case SET combine — codex32 `interpolate_at` does raw case-sensitive cross-share compares on hrp (lib.rs:241-243 → `MismatchedHrp("MS","ms")`) and id (:250-252). In `combine_shares`, AFTER each share passes its first `Codex32String::from_string(s.clone())` (preserving the within-one-string mixed-case `InvalidCase` rejection — NEVER lowercase before the first parse), re-canonicalize: `Codex32String::from_string(c.to_string().to_ascii_lowercase())` (the checksum engine case-folds, so it re-parses — infallible in practice, probe-proven byte-identical; still route the Result via `?` through `Error::Codex32`, R0-r2 M1-r2) and pass the CANONICAL vector to both field extraction and `interpolate_at` (also makes the recovered output lowercase — codex32 re-emits uppercase when hrp is uppercase).
- **`decode_with_correction` clean-codeword pass-through (R0-r1 I2):** `decode.rs:232-237`'s residue==0 path calls `decode(s)` with the ORIGINAL string — covered transitively via discriminate, but it is a distinct public surface: today a PRISTINE uppercase card fails while a CORRUPTED one repairs fine (corrections re-emit lowercase). U6 pins it.
- **Inspect nuance — CLOSED (R0-r1 M2):** codex32 `Parts::data()` case-folds (Fe::from_char via CHARS_INV maps both cases) — inspect.rs:75 + shares.rs:240 need NO rerouting. Inspect's "surface the raw observation" intent is about non-table tag VALUES, not case (codex32 enforces whole-string uniform case) — lowercasing loses no diagnostic information, and transitively fixes ms-cli `analyze()`'s spurious wrong-hrp/share-index reasons for uppercase cards (cmd/inspect.rs:115/:123).
- Encode/emission stays lowercase everywhere (untouched). Mixed-case behavior unchanged (codex32 `InvalidCase` fires before any of this).
- Error-display note: `WrongHrp { got }` keeps the RAW-observed casing? No — after canonicalization a true wrong-HRP (e.g. `XS1…`) reports the lowercased `xs` — acceptable and consistent (the message names what it compared); note in CHANGELOG.

## Tests (TDD red-first)

- **U1:** `decode(&valid_ms1.to_uppercase())` returns the SAME (Tag, Payload) as the lowercase twin (today: RED with `WrongHrp{got:"MS"}`).
- **U2:** `inspect(&upper)` report equals the lowercase report (tag, threshold, share index, payload fields).
- **U3:** `combine_shares(&[upper shares...])` recovers the same secret as lowercase (existing K-of-N fixture, uppercased — likely ALREADY GREEN today per R0-r1 M3: digit threshold, Fe folds, uniform set passes the raw compares — keep as characterization); MIXED-set (one uppercase share among lowercase) WORKS under the C1(a) combine-side canonicalization (RED today: codex32 `MismatchedHrp`). **U3-guard (R0-r1 I1 + R0-r2 I1-r2, SECURITY):** fixture MUST be the UNIFORM-UPPERCASE SAME-ID hand-built pair — `from_seed(HRP, 2, "tst7", Fe::S, wire_bytes)` + `from_seed(HRP, 2, "tst7", Fe::A, filler_of_same_len)`, `.to_uppercase()` BOTH (codex32's interpolate_at runs its length/hrp/threshold/id validation loop over ALL shares BEFORE the index-match short-circuit — a lowercase companion dies in MismatchedHrp and does NOT reproduce the leak; empirically proven at 952bebd: the uniform-uppercase pair returns `Ok((ENTR, Entr(secret)))` TODAY — the realistic whole-card-set-uppercased configuration). RED = `Ok` carrying the exact secret payload; post-fix = `Err(SecretShareSuppliedToCombine)` (the canonicalized fields make the shares.rs:207 guard see `b's'`).
- **U4:** mixed-case WITHIN one string still rejects (`InvalidCase` via codex32) — re-pin.
- **U5:** true wrong-HRP uppercase (`XS1…`) → `WrongHrp { got: "xs" }` (the canonicalized report). Fixture: `from_seed("xs",…).to_string().to_uppercase()` — `from_seed("XS",…)` cannot mint it (its internal set_check_case rejects the mixed intermediate; R0-r2 M4-r2).
- **U6 (R0-r1 I2):** `decode_with_correction(&upper_clean)` == the lowercase twin's result with EMPTY corrections (RED today); `decode_with_correction(&upper_with_1_error)` repairs (GREEN today — pins the pristine-fails/corrupted-repairs asymmetry's resolution).
- **CLI cell (R0-r1 I3):** one ms-cli integration cell (`ms decode <UPPERCASE_MS1>` succeeds, crates/ms-cli/tests/) — CI runs ONLY `cargo test -p ms-cli` (no Windows lanes; ms-codec cells never execute in CI — `ms-codec-no-ci-workflow` is an open FOLLOWUP), so this is the sole CI-executed coverage; the local full suite + clippy is the actual gate (I9 precedent).
- Existing suite green (the toolkit's staged characterization cells are TOOLKIT-side and invert at its pin bump — not this repo's concern beyond the FOLLOWUPS note). Per-cell red/green annotated above (R0-r1 M3). No test pins raw-case `WrongHrp.got` (M7 sweep clean).
